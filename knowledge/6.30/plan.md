# 前処理・特徴量抽出 実装方針

第4回（131-004）・第5回（131-005）ゼミ資料に基づく。

---

## 1. 使用データ

アプリが出力する以下の CSV ファイルを使用する。

| ファイル名     | 役割                                             |
| -------------- | ------------------------------------------------ |
| `acc.csv`      | 加速度の時系列ログ（特徴量抽出の元データ）       |
| `dansa.csv`    | 段差イベント記録（段差ラベル作成に使用）         |
| `dekoboko.csv` | でこぼこイベント記録（でこぼこラベル作成に使用） |

### acc.csv のカラム

| カラム名    | 内容                         |
| ----------- | ---------------------------- |
| `timestamp` | データ取得時刻               |
| `elapsed`   | 計測開始からの経過時間（ms） |
| `accX`      | X 軸加速度                   |
| `accY`      | Y 軸加速度                   |
| `accZ`      | Z 軸加速度                   |

- サンプリング間隔：250 ms（4 Hz）
- 解析対象：主に `accZ`（鉛直方向）

> 注：ゼミ資料（131-004）では `acclogger.csv` と記載されているが、実際のファイル名は `acc.csv`。

### イベント記録 CSV のカラム（dansa.csv / dekoboko.csv 共通）

| カラム名                | 内容                         |
| ----------------------- | ---------------------------- |
| `timestamp`             | イベント記録時刻             |
| `elapsed_ms`            | 計測開始からの経過時間（ms） |
| `acc_x / acc_y / acc_z` | イベント記録時点の各軸加速度 |
| `latitude / longitude`  | イベント記録時点の位置情報   |
| `event`                 | イベント種別                 |

`event` の値：

- `dansa_start` / `dansa_stop`
- `dekoboko_start` / `dekoboko_stop`

## 2. ウィンドウ分割

| パラメータ               | 値                 |
| ------------------------ | ------------------ |
| ウィンドウ幅             | 1.0 秒             |
| スライド幅               | 0.25 秒            |
| 1 ウィンドウのサンプル数 | 4 点（250 ms × 4） |

先行研究（三浦さん）の指標計算が実質 1.0 秒窓を対象としていたことに対応。段差のような瞬間的イベントがウィンドウ境界にかかっても特徴が消えないよう、0.25 秒スライドを採用。

---

## 3. ラベル付け

付与するラベルは `flat` / `dansa` / `dekoboko` の 3 クラス。

スクリプトで自動付与した後、加速度波形・イベント記録・ラベルを同一時間軸で可視化して目視確認する。目視で明らかに誤りと判断したウィンドウは学習・評価対象から除外する（`unknown` クラスは学習には使わない）。

### 3-1. でこぼこのラベル付け

`dekoboko_start` ～ `dekoboko_stop` をでこぼこ区間とする。

各ウィンドウについて、でこぼこ区間との重なり割合が以下を満たす場合に `dekoboko` を付与：

```
T_overlap / T_win >= 0.5
```

T_win = 1.0 s なので、0.5 秒以上重なるウィンドウを `dekoboko` とする。わずかに重なっただけの曖昧なウィンドウを誤ってラベル付けすることを防ぐ。

### 3-2. 段差のラベル付け

`dansa_start` ～ `dansa_stop` を段差区間とする。でこぼこと同じ方法で付与する。

各ウィンドウについて、段差区間との重なり割合が以下を満たす場合に `dansa` を付与：

```
T_overlap / T_win >= 0.5
```

T_win = 1.0 s なので、0.5 秒以上重なるウィンドウを `dansa` とする。

### 3-3. 優先度（重複時の扱い）

`dansa` と `dekoboko` の両条件を満たすウィンドウは **`dansa` を優先**する。

付与の優先順位：`dansa` > `dekoboko` > `flat`

### 3-4. flat のラベル付け

`dansa` にも `dekoboko` にも該当しないウィンドウを `flat` とする。

## 4. 特徴量抽出

初期検証では Z 軸加速度（`accZ`）のみから抽出。X 軸・Y 軸・合成加速度は今後の拡張。

1 ウィンドウ内の Z 軸加速度を `z_1, z_2, ..., z_N`（N=4）とする。

| 特徴量       | 定義                           | 目的                                  |
| ------------ | ------------------------------ | ------------------------------------- | --- | ---------------------------------------- |
| `z_range`    | `max(z_i) - min(z_i)`          | ウィンドウ内の振れ幅（常に ≥ 0）      |
| `z_diff_max` | `max(                          | z*i - z*{i-1}                         | )`  | 隣接サンプル間の最大変化量（急激な変化） |
| `z_rms`      | `sqrt(mean(z_i^2))`            | 振動の大きさ（先行研究の RMS と対応） |
| `z_std`      | `sqrt(mean((z_i - z_mean)^2))` | 振動のばらつき                        |

**補足：**

- `z_range`：振れ幅なので絶対値不要（max - min は常に非負）
- `z_diff_max`：`z_range` が区間全体の幅なのに対し、こちらは隣接 2 点間の最大変化量。段差の「鋭さ」に対応
- `z_rms`：先行研究の `a_RMS(t)` と同じ計算式

## 5. データセットの構造

1 ウィンドウ = 1 行。

| カラム       | 内容                          |
| ------------ | ----------------------------- |
| `run_id`     | 走行回の識別子（例：`run01`） |
| `window_id`  | ウィンドウの通し番号          |
| `start_ms`   | ウィンドウ開始時刻（ms）      |
| `end_ms`     | ウィンドウ終了時刻（ms）      |
| `label`      | `flat` / `dansa` / `dekoboko` |
| `z_range`    | 特徴量                        |
| `z_diff_max` | 特徴量                        |
| `z_rms`      | 特徴量                        |
| `z_std`      | 特徴量                        |

`run_id` / `window_id` / `start_ms` / `end_ms` はモデルに入力しない。誤分類サンプルを波形に戻って確認するための管理情報として保持する。

---

## 6. 学習・評価の分割

**走行回（run）単位で分割する。ウィンドウ単位のランダム分割は行わない。**

理由：0.25 秒スライドにより隣接ウィンドウは非常に似た波形になる。ウィンドウ単位で分割すると、ほぼ同じ波形が学習・評価の両方に入り、性能を過大評価する。

---

## 7. スマートフォン座標系

- 計測前に水平器機能を使い、スマートフォンができるだけ水平になるよう固定する
- 固定位置・向きを計測ごとに統一する
- これにより Z 軸 ≈ 鉛直方向として扱う

走行中の傾きや段差通過時の姿勢変化は完全補正できない。実測後に影響が大きければ回転ベクトルを用いた座標変換を次段階の課題として検討する。
